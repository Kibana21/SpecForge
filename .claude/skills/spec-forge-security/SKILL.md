# SPEC FORGE Security Skill

This skill encodes the security patterns SPEC FORGE follows across its FastAPI backend and Next.js frontend. The goal is that security-sensitive code generated by Claude is correct by construction — not "mostly right with a few subtle vulnerabilities."

## Stack assumptions

- **Backend**: FastAPI, Pydantic v2, SQLAlchemy (async), `passlib[bcrypt]`, `python-jose` or `pyjwt` for JWTs, `slowapi` for rate limiting, Redis for token blocklist and rate counters.
- **Frontend**: Next.js (App Router), TypeScript, server actions/route handlers for API proxy where applicable.
- **Infra**: Deployed behind HTTPS terminator (reverse proxy / platform). Secrets via env vars (local `.env`, platform secret store in prod).

If you're working on code outside these assumptions, flag the deviation before applying these patterns blindly.

## How to use this skill

Before writing security-sensitive code, **read the relevant reference file**:

- **`references/jwt-auth.md`** — JWT issuance/validation, refresh rotation, algorithm pinning, kill-switch (JTI blocklist), login/logout/password-reset/email-verification flows, account lockout, enumeration prevention.
- **`references/fastapi-patterns.md`** — FastAPI dependencies for auth/RBAC, CORS, rate limiting, Pydantic validation patterns, security headers middleware, SQLAlchemy safe-query patterns, file upload hardening, audit logging.
- **`references/nextjs-patterns.md`** — Token storage (memory + httpOnly cookie split), CSRF defense (SameSite + double-submit), CSP, secure cookie flags, fetch wrappers, server-side session validation, client/server boundary rules.
- **`references/anti-patterns.md`** — Specific code patterns to refuse or rewrite. Read this before generating any auth code.

You don't need to read all four every time. Match the file to the task: writing a login endpoint → `jwt-auth.md` + `fastapi-patterns.md`. Building a Next.js protected route → `nextjs-patterns.md`. Adding a new RBAC-protected endpoint → `fastapi-patterns.md`.

## Inviolable principles

These are non-negotiable. If a request asks you to violate one, refuse and explain why.

1. **No hardcoded secrets, ever.** Secrets come from env vars. If a secret would need to be added inline to make code work, stop and ask.
2. **Authorization is checked server-side on every protected endpoint.** Frontend checks are UX only. There is no "trusted client."
3. **Every authenticated endpoint runs through the `require_user()` or `require_role(...)` dependency.** No ad-hoc token parsing inside route handlers.
4. **JWT decode always pins the algorithm.** `jwt.decode(token, secret, algorithms=["HS256"])` — never let the library accept whatever's in the header.
5. **SQL is always parameterized via SQLAlchemy.** No f-string SQL, no `.format()` SQL, no string concatenation into queries.
6. **No PII, passwords, tokens, or secrets in logs.** Log `user_id` and `correlation_id`, not emails or token values.
7. **Errors visible to clients are generic.** Detailed errors go to server logs. Stack traces never reach the browser in production.
8. **Auth events are audit-logged.** Login (success/fail), logout, refresh, password change, role change, permission denial — all logged with `user_id`, `ip`, `user_agent`, `correlation_id`, `timestamp`.

## Pre-flight checklist

Before returning any code that touches auth, sessions, user data, or external services, run through this mentally. If you can't check a box, fix it before responding.

- [ ] **Secret source**: every secret read from env via a settings module — no literals.
- [ ] **Auth dependency**: every protected endpoint has `Depends(require_user)` or `Depends(require_role(...))`.
- [ ] **Authorization check**: for resource access, ownership/membership verified — not just role.
- [ ] **Algorithm pinned**: JWT decode specifies `algorithms=[...]` explicitly.
- [ ] **Response schema**: Pydantic response model excludes password hashes, tokens, internal IDs, sensitive flags.
- [ ] **Input validation**: request body is a Pydantic model with explicit types and constraints.
- [ ] **SQL**: queries use ORM or `text()` with bound params — no string interpolation of user input.
- [ ] **Status codes**: 401 for missing/invalid auth, 403 for authenticated-but-unauthorized, 404 to avoid leaking existence to unauthorized users.
- [ ] **Errors**: generic message to client, detailed log server-side, no stack trace leakage.
- [ ] **Logging**: no passwords, tokens, full emails, or PII in log lines. `correlation_id` included.
- [ ] **Audit event**: if this is an auth action (login, refresh, role change, permission denial), emit an audit log entry.
- [ ] **Rate limit**: auth endpoints, password reset, signup, and any expensive operation have a `slowapi` limit applied.
- [ ] **CORS**: no `allow_origins=["*"]` — uses `FRONTEND_URL`.
- [ ] **Cookies**: `httponly=True, secure=True, samesite="lax"` (or `"strict"` where appropriate).
- [ ] **CSRF**: for cookie-authenticated state-changing routes, CSRF defense is present (SameSite + double-submit token or origin check).
- [ ] **No `eval` / `exec` / `shell=True`** on user-controlled input.
- [ ] **Dependencies pinned**: any new dep is added to `requirements.txt` with `==` and noted for lockfile update.

## Anti-patterns to refuse or flag

These appear frequently in generated code and are wrong. If a user asks for them, push back.

| Anti-pattern | Why it's wrong | Correct approach |
|---|---|---|
| `jwt.decode(token, secret)` without `algorithms=[...]` | Allows algorithm confusion attacks (incl. `alg: none`) | Always pass `algorithms=["HS256"]` |
| `allow_origins=["*"]` with `allow_credentials=True` | CORS spec violation; browsers reject, but worse — if accepted it's a CSRF bypass | Whitelist exact origin from env |
| Storing JWT in `localStorage` | Readable by any XSS payload | Access token in React memory; refresh token in httpOnly cookie |
| `f"SELECT * FROM users WHERE email = '{email}'"` | SQL injection | `select(User).where(User.email == email)` |
| Returning `{"error": str(e)}` from a `try/except` | Leaks internal details (paths, stack, lib versions) | Log full error server-side; return generic message |
| Same response for "user not found" vs "wrong password" being different | Email enumeration | Identical generic response for both branches |
| `if user.role == "admin"` inline inside a handler | Bypassable, not centralized, not audit-logged | `Depends(require_role("admin"))` |
| `password = request.password; logger.info(f"login attempt {request}")` | Logs the password | Never log the request object directly; log specific safe fields |
| `os.system(f"convert {filename} ...")` | Shell injection via filename | Use library APIs; if shell unavoidable, `subprocess.run([...], shell=False)` |
| `open(f"/uploads/{user_filename}")` | Path traversal via `../` | Generate random filename, never trust user-supplied name for path |
| `verify=False` on `requests` / `httpx` | Disables TLS validation | Never. If cert issues, fix the cert |
| Hardcoded `JWT_SECRET = "changeme"` | Secret in source control | Read from env; fail loudly at startup if missing |
| Catching `Exception` and swallowing | Hides real errors; can mask auth failures | Catch specific exceptions; let unexpected ones propagate to error middleware |
| Frontend-only role gating with no backend check | Trivially bypassed via direct API call | Backend `require_role()` is the source of truth; frontend gating is UX only |

## Response codes — quick reference

- **200 / 201** — Success.
- **400** — Malformed request (validation failure). Pydantic returns this automatically.
- **401 Unauthorized** — No token, expired token, invalid signature, revoked refresh token. Caller is not authenticated.
- **403 Forbidden** — Authenticated but lacks permission. Use sparingly — prefer 404 when leaking existence is itself a problem.
- **404 Not Found** — Resource doesn't exist OR exists but the requester isn't authorized to know. Default to 404 for "you can't see this" unless there's a UX reason to differentiate.
- **409 Conflict** — Duplicate (e.g., signup with existing email — but be careful, this leaks enumeration; see `references/jwt-auth.md`).
- **422** — Pydantic validation error (FastAPI default).
- **429 Too Many Requests** — Rate limit hit.
- **500** — Internal error. Generic message to client, full detail to server logs.

## When the user asks for something risky

If a user requests behavior that violates these standards (e.g., "just disable CORS for now", "put the JWT secret in the code so it works", "skip the rate limit, it's annoying in dev"), respond with:

1. Acknowledge the immediate friction they're hitting.
2. Explain the specific risk in one or two sentences.
3. Offer the correct alternative that solves their actual problem (e.g., dev-only env var, separate dev config, higher rate limit for local IPs).

Don't lecture, but don't comply silently either. The user has compliance and audit obligations — protecting them from a foot-gun is part of the job.

## What this skill does NOT cover

- Infrastructure security (VPC, WAF, DDoS) — that's platform/ops layer.
- Cryptographic primitives — use vetted libraries (`passlib`, `cryptography`, `pyjwt`), never roll your own.
- Compliance frameworks (MAS, GDPR, SOC2) at a policy level — this skill captures engineering controls that *support* compliance, but mapping to specific clauses is a separate exercise.
- Penetration testing or threat modeling methodology.

If a request touches these areas, name the gap and suggest the user consult the appropriate resource.
