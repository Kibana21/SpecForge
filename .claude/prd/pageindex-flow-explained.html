01The mental model

PageIndex is a vectorless, reasoning‑based document index.

Most RAG systems chop a document into fixed‑size chunks, embed them as vectors, and retrieve by cosine similarity. PageIndex does something different: it turns each document into a tree of sections — like a generated, machine‑readable table of contents where every node knows its title, a one‑line summary, and the page range it covers. Retrieval is then an LLM navigating that tree to pick the relevant sections, rather than a vector nearest‑neighbour lookup.

In SpecForge's E2 project zone there is no vector fallback — the PageIndex tree is the index. That single tree per document does three jobs:

① Retrieval substrate

The reasoning tree the LLM searches to answer "which sections are relevant to this query?" — feeding the Requirement Understanding and every downstream artifact.

② Concept grounding

The outline (node_id · title · summary lines) is handed to the wiki compiler so each emergent concept can cite the exact sections it was synthesised from.

③ Traceability spine

Node references (S:<doc>:<node> tokens) become the "Grounded in" chips and the provenance lineage you click in the UI.

The provider seam

Everything sits behind CorpusIndexProvider (corpus_index/base.py) with build_index() + tree_search() — so it's swappable and fully mockable.

Why a tree, not chunks? Requirements documents are hierarchical (sections, sub‑clauses). A tree preserves that structure, so the model retrieves a coherent "§4.2 Data Retention" rather than three half‑sentences that happen to be near a query vector — and it can cite that section back to the user.

03Stage 1 — Upload & the synchronous parse

When a user uploads a project source, the request handler does the cheap, durable work synchronously and pushes the expensive work to a background worker. The crucial detail: extracted_text is set here (so the requirement‑extractor can use it instantly), while the PageIndex tree build is dispatched async.

# app/api/documents.py  (upload handler, ~L88–115)
doc = Document(..., parse_status="pending", indexing_status="pending")
db.add(doc); await db.flush()

# Synchronous text extraction — used immediately by requirement_extractor
text = parser.parse(content, mime)
doc.extracted_text = text
doc.parse_status = "done"
await db.commit()

# Dispatch PageIndex ingestion (builds the reasoning tree → document_trees).
# Best-effort: a broker outage must not fail the (already durable) upload.
from workers.dispatch import dispatch
from workers.tasks import ingest_project_source
dispatch(ingest_project_source, str(doc.id))
Why dispatch() and not .delay()? workers/dispatch.py does a fast TCP pre‑flight against Redis so the API never hangs if the broker is down. The upload is already committed and durable; ingestion is re‑triggerable. (See backend/CLAUDE.md → Celery gotchas.)

04Stage 2 — The ingestion task

_ingest_project_source(document_id) (workers/tasks.py:1272) is the orchestrator. It decides what bytes to feed PageIndex based on MIME type, calls the provider, stores the result, and chains the wiki compile.

Choosing the index input

Document typeWhat gets indexedWhy
application/pdfRaw PDF bytes → PageIndex native pathPageIndex uses an LLM to infer section structure directly from the PDF — smarter than markdown on flat reports.
DOCX / images (RICH_MIMES)Azure‑converted markdown (cached in extracted_text)No native PageIndex path; the markdown carries the headings PageIndex needs.
TXT / MDextracted_text as markdown bytesAlready plain text from the sync upload phase.
# workers/tasks.py  _ingest_project_source  (~L1305–1358)
if doc.mime_type in RICH_MIMES:
    if not doc.extracted_text:                 # convert once, commit separately
        markdown = await md_provider.convert(data, doc.mime_type, doc.filename)
        doc.extracted_text = markdown; await db.commit()  # so a tree-build fail can't roll back Azure
    if doc.mime_type == "application/pdf":
        index_data, index_content_type = data, "application/pdf"
    else:
        index_data, index_content_type = doc.extracted_text.encode(), "text/markdown"
else:                                            # TXT / MD
    index_data, index_content_type = (doc.extracted_text or "").encode(), "text/markdown"

result = await get_corpus_index_provider().build_index(
    data=index_data, content_type=index_content_type, filename=doc.filename)

await db.execute(delete(DocumentTree).where(DocumentTree.document_id == doc.id))
db.add(DocumentTree(document_id=doc.id, project_id=doc.project_id,
                    tree_json=result.tree, page_texts=result.page_texts,
                    node_count=result.node_count, model=result.model))
doc.indexing_status = "done"; await db.commit()
compile_project_wiki_for_doc.delay(project_id_str, document_id)   # → Stage 5
Resilience built in. Markdown conversion is committed before the tree build, so a PageIndex failure can't waste the expensive Azure call on retry. On any error the task flips indexing_status="error" with the message (or detects a deleted doc) instead of crashing.

05Stage 3 — Building the tree (the PageIndex provider)

PageIndexProvider.build_index() (corpus_index/pageindex_provider.py:146) is where the vendored PageIndex OSS actually runs. Three things happen: LiteLLM is pointed at Vertex, the right builder is invoked, and per‑page text is captured for later node‑text fetching.

① Point LiteLLM at our Vertex Gemini

PageIndex internally calls litellm.completion(model="vertex_ai/..."). We configure it to use SpecForge's service account and project — verified in the opt‑in T6 spike (tests/test_pageindex_vertex.py).

# _configure_litellm()  (L37)
litellm.drop_params = True                          # tolerate unsupported OpenAI params
litellm.vertex_project  = settings.gemini_project_id
litellm.vertex_location = settings.gemini_location
os.environ["GOOGLE_APPLICATION_CREDENTIALS"] = settings.google_service_account_path
return settings.pageindex_model               # "vertex_ai/gemini-3.5-flash"

② Invoke the right builder

InputBuilder (vendor/pageindex)Mode
PDFpage_index_main(path, opt)sync — LLM reads the PDF's table of contents / page ranges
Markdownmd_to_tree(...)async — tree from # headings

Both run inside a thread executor (Celery‑safe) and return a structure of nodes. We pass if_add_node_summary=yes, if_add_node_id=yes, if_add_node_text=no — we want the outline (id, title, summary, page range) but fetch body text ourselves from page_texts.

③ Normalise + the heading‑less fallback

PageIndex output is coerced into the canonical {"nodes":[...]} shape. The important safety net: PageIndex builds nodes from document structure. A flat, heading‑less press release yields zero nodes — which in the E2 zone (no vector fallback) would make the doc invisible. So we synthesise one node per page:

# build_index() tail  (L177–187)
tree = _normalize_tree(raw)
if count_nodes(tree) == 0 and page_texts:
    tree = _fallback_tree(page_texts, filename)   # one node per page → still retrievable
    log.info("pageindex_fallback filename=%s nodes=%d", filename, count_nodes(tree))
return DocTree(tree=tree, page_texts=page_texts,
               node_count=count_nodes(tree), model=model)

Per‑page text is captured up front — for PDFs via PyMuPDF (_pdf_page_texts), for markdown by paginating into 1500‑char pages (_paginate). This page_texts map is what lets us later fetch a node's full body from its page range without storing duplicate text in every node.

All PageIndex imports are lazy. The module loads even when the package isn't installed, so the mock path (CI, tests) stays usable with zero Vertex/network calls.

The result lands in one DocumentTree row (models/project_source.py): unique per document_id, FK‑cascaded to both the document and the project.

What a node looks like

Each node is the unit of retrieval and citation. start_index/end_index are page numbers into page_texts.

nodes:
  node_id "0000" · "Secure Document Portal"  (pp 1–4)
  summary: "Overview of the portal's purpose and scope…"
  nodes:
    node_id "0001" · "Document Management & Processing" (pp 1–1)
    summary: "Upload, processing, version control of files…"
    node_id "0002" · "Security, Authentication & Compliance" (pp 2–3)
    summary: "MFA, RBAC, AES‑256, GDPR/SOC2 on Azure…"

The helpers in corpus_index/base.py operate on this shape everywhere: iter_nodes (depth‑first walk), find_node (by id), node_text (concatenate the page range, fall back to a node‑level text field), count_nodes, and flatten_outline (render the LLM‑readable outline for search).

One core, two wikis. compile_doc_into_wiki (wiki/compile_core.py) is a parameterised copy of the App Brain pipeline, bound to a concrete wiki by a WikiScope (which models + which scope/doc columns). The Project Wiki passes ProjectWikiSummary/ProjectWikiConcept + project_id; the App Brain keeps its own untouched copy ("copy, not refactor").

09Stage 7 — Retrieval: reasoning over the tree

Concept compilation hands the LLM the whole outline. The other use of the tree is targeted retrieval — answering "which sections matter for this query?" — used by the Requirement Understanding, the clarifier, and every downstream artifact via depth_search (context/docs_layer.py:154).

# PageIndexProvider.tree_search  (pageindex_provider.py:189)
outline = flatten_outline(docs)                 # [D0] node_id · title — summary, …
result  = await run_tree_search(query, outline, top_k)   # DSPy module reasons, picks node_ids
for sel in result["selections"][:top_k]:
    node = find_node(doc.tree, sel["node_id"])
    out.append(RetrievedSection(
        node_id=node["node_id"], title=node["title"],
        page_start=node["start_index"], page_end=node["end_index"],
        text=node_text(node, doc.page_texts),    # body fetched from page_texts
        reason=sel["reason"]))

So retrieval is: flatten every in‑scope tree → LLM selects node_ids with reasons → fetch each node's body from page_texts. No embeddings, no vector store. top_k defaults to tree_search_top_k = 8.

Breadth vs depth. docs_layer.py exposes two views over the same trees: breadth (the full outline inventory of every document, so generation never misses a doc) and depth (tree_search for a specific query/unit). The RU grounding uses both.

10Stage 8 — Grounding & the trace chips you click

This is where the tree closes the loop back to the UI. Two paths consume the stored grounding:

Concept "Grounded in" chips

Each concept's tree_node_refs render as section chips. Clicking one calls GET /projects/{id}/sources/{doc}/section/{node}, which runs find_node + node_text over the stored tree and returns the exact section text + page range.

Clarification & RU citations

The clarifier and RU emit canonical tokens — S:<doc>:<node> (source section), C:<slug> (wiki concept), F:<fact> — that IntakeTraceChip resolves to a floating popover. The S: resolver is the same section endpoint, powered by the PageIndex tree.

# app/api/project_wiki.py — section drill-down (the S: resolver)
node = find_node(tree.tree_json, node_id)
return ok({"doc_name": doc_name, "title": node["title"],
            "pages": f"{start}-{end}",
            "text": node_text(node, tree.page_texts)})   # ← page_texts again

And the provenance lineage (services/provenance/lineage.py) walks Documents → Concepts using exactly these tree_node_refs (a concept's source doc_ids come from source_doc_ids ∪ tree_node_refs doc_ids), so the LineageMap is, at root, a view over the PageIndex grounding.

11Mock vs production

The whole flow runs deterministically with zero network when LLM_PROVIDER=mock — that's what keeps CI green and tests fast. The provider factory picks the implementation:

# corpus_index/__init__.py
if choice == "auto":
    choice = "mock" if settings.llm_provider == "mock" else "pageindex"
StepProduction (PageIndex)Mock
Tree buildLiteLLM → Vertex Gemini reasons over structureMockCorpusIndexProvider: one node per 1500‑char page under a single root 0000 — same shape, no LLM
Summary / plan / page3 DSPy ChainOfThought modulesDeterministic dict in compile_doc_into_wiki: an overview concept, tree_node_refs = first valid node
tree_searchDSPy run_tree_search picks node_idsMock provider returns deterministic selections

Because the mock produces the same tree shape and the same DB rows, every stage downstream (concepts, refs, retrieval, trace chips) works identically — only the intelligence is stubbed.

One‑sentence summary. A project document is parsed, then PageIndex (via Vertex Gemini) turns it into a tree of titled, page‑ranged sections stored as a DocumentTree; that tree is the retrieval substrate and the grounding that the wiki compiler cites when it synthesises emergent concepts — and those validated node references are exactly the source‑section citations you click in the Understanding Studio.
